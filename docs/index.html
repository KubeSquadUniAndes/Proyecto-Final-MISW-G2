TravelHub

Plataforma de reservas y hospedajes con arquitectura de microservicios desplegada en AWS EKS, aplicaciones móvil Android y web Angular.

01

Visión General

Contexto del sistema

TravelHub es una plataforma que conecta viajeros y hoteles para la gestión de reservas y hospedajes. Está compuesta por una aplicación móvil Android, un frontend web y un backend de microservicios.

Los actores principales son el Usuario Viajero (accede vía app móvil o web) y el Usuario Hotel / Administrador (gestiona tarifas, habitaciones y políticas).

Decisiones arquitectónicas clave

• Microservicios desplegados en AWS EKS con escalado por pods (k8sPod).

• Istio Ambient Mode (sidecarless): service mesh con ztunnel (mTLS L4) y waypoint proxy (políticas L7) sin inyección de sidecars.

• Separación de lectura y escritura: escritura transaccional en PostgreSQL.

• Cifrado de datos sensibles con extensión pgcrypto AES-256 en PostgreSQL.

• Secretos gestionados en AWS Secrets Manager con sincronización automática vía External Secrets Operator.

• Arquitectura hexagonal (puertos y adaptadores) en cada microservicio para independencia de infraestructura.

Meta de disponibilidad: RPO ≤ 5 min · RTO ≤ 15 min — garantizados mediante disponibilidad PostgreSQL.

Diagrama de Componentes

Frontend

Presentación Móvil

Arquitectura MVC en Android. La Vista delega acciones al Controlador que consume los servicios REST del ApiGateway. Desarrollada en Kotlin.

Presentación Web

SPA en Angular con arquitectura de componentes. El FrontHub se despliega en AWS CloudFront como CDN. Vista → Controlador → HTTP al ApiGateway.

Servicios de negocio

Reservas
k8sPod · count 1..3 · FastAPI
Gestión del ciclo de vida completo de reservas: creación, validación de disponibilidad, cambio de estado y finalización.
Hospedajes
k8sPod · count 1..3 · FastAPI
Administración de hoteles, habitaciones, tarifas y políticas. Expone endpoints para gestión por parte del rol administrador/hotel.
Pagos
k8sPod · count 1..2 · FastAPI
Integración con proveedores de pago externos a través del GateWay intermediario. Maneja estados de transacción y registro de pagos.
Usuarios
k8sPod · count 1 · FastAPI
Registro, autenticación y perfil de usuarios. Trabaja en conjunto con LoginHandler para validación de tokens.
Notificaciones
AWS Lambda · count 0..3
Envío de notificaciones push a dispositivos móviles. Suscrito a eventos del Message Broker para notificar cambios de estado de reservas.

Componentes transversales

LoginHandler

Validación de tokens JWT. Intermediario entre el ApiGateway y los servicios internos. Conecta con Redis SessionCache vía TCP/IP.

Monitor

Supervisión del estado de los pods del cluster K8s. Coordina con TimeoutDetector y DetectorAnomalias para alertas proactivas.

DetectorAnomalias

Análisis de patrones anómalos en el cluster. Componente k8sPod que interactúa con el Monitor y publica eventos al Message Broker.

Diagrama de Concurrencia

El sistema gestiona la concurrencia a través de tres mecanismos principales: escalado horizontal de pods, separación lectura/escritura, y mensajería asíncrona.

Flujo de escritura (transaccional)

Las operaciones de escritura pasan por el ApiGateway → Reservas/Hospedajes/Pagos → PostgreSQL. Cada transacción es atómica y escribe con write(transaction).

Al completarse, se publica el evento RoomAvailabilityChanged al Message Broker de forma asíncrona.

Contenedores

Todos los microservicios se empaquetan como imágenes Docker construidas en el pipeline CD y almacenadas en AWS ECR. El despliegue se realiza con kubectl rollout sin downtime.

Service Mesh — Istio Ambient

Modo Ambient (Sidecarless)

TravelHub usa Istio en modo ambient, eliminando la necesidad de inyectar sidecars en cada pod. La malla se divide en dos capas independientes: el proxy ztunnel a nivel de nodo (L4) y el proxy waypoint para políticas L7 por namespace.

El namespace workloads está etiquetado con istio.io/dataplane-mode: ambient. Todos los servicios apuntan al waypoint con istio.io/use-waypoint: waypoint-private.

Componentes del plano de datos

ztunnel — DaemonSet por nodo. Intercepta todo el tráfico TCP, provee mTLS transparente con identidades SPIFFE/SVID basadas en ServiceAccounts, telemetría L4 y reenvío seguro vía protocolo HBONE.

waypoint-private — Proxy Envoy dedicado (Gateway API, clase istio-waypoint, puerto 15008). Aplica políticas L7: AuthorizationPolicies, DestinationRules, retries y rate limiting antes de entregar al pod destino.

mTLS automático (ztunnel)

Todo el tráfico este-oeste entre pods del namespace workloads viaja cifrado con mTLS mutuo. El ztunnel establece identidades SPIFFE/SVID basadas en los ServiceAccounts de Kubernetes —sin configuración adicional— ambient mode lo habilita automáticamente para todos los pods en el namespace.

Para AWS: los microservicios usan IRSA (IAM Roles for Service Accounts) en lugar de EC2 IMDS, evitando conflictos con el tráfico interceptado por ztunnel.

Waypoint — políticas L7 (02-waypoint.yaml)

El waypoint waypoint-private (HBONE port 15008) es el único punto donde se evalúan políticas L7. Recibe tráfico del ztunnel del nodo destino, aplica AuthorizationPolicies, DestinationRules (outlier detection, connection pools) y rate limits, luego reenvía al pod final.

Subsets definidos: hospedajes-v1 y users-v1 —preparados para canary deployments y traffic shifting.

Authorization Policies — mínimo privilegio (11-authorization-policies.yaml)

El ServiceAccount login-handler-sa tiene una política DENY que impide al microservicio login-handler llamar a detector-anomalias, notificaciones, reservas y hospedajes. login-handler solo puede comunicarse con users-service, aplicando el principio de mínimo privilegio a nivel de service mesh sin modificar código de aplicación.

Rate Limiting local (13-rate-limit.yaml)

EnvoyFilter aplicado al pod reservas-app: 100 req/s sostenidas con burst de 500 tokens (capacidad 5×). Rate limiting local por instancia —cada réplica mantiene su propio bucket de tokens. Cuando se excede el límite el proxy responde HTTP 429 con header x-local-rate-limit: true.

Access Logging mesh-wide (16-istio-access-logging.yaml): El recurso Telemetry en istio-system habilita logging HTTP via proveedor envoy para todos los proxies de la malla — ztunnel y waypoint registran método, URL, status code, latencia y bytes en cada salto de tráfico.

Pipeline de logs — Grafana Alloy (15-alloy.yaml)

Grafana Alloy se despliega en el namespace istio-system con un único deployment (no DaemonSet). Accede a los logs de todos los pods del cluster vía la API de Kubernetes (loki.source.kubernetes), evitando montaje de volúmenes en los nodos.

Labels promovidos a Loki: namespace, app, container, node. Label externo: cluster=travelhub-prod. Batching de 5 s / 512 KiB.

Seguridad: uid 473 (non-root), filesystem de solo lectura, sin capabilities adicionales. Node selector: role: observability.

Métricas — Prometheus & Grafana

Prometheus recolecta las métricas estándar de Istio expuestas por ztunnel y waypoint. Métricas clave disponibles:

• istio_requests_total — total de peticiones HTTP por servicio, código y método.
• istio_request_duration_milliseconds — latencia p50/p90/p99.
• istio_tcp_sent_bytes_total — bytes TCP transmitidos por ztunnel.

Dashboards Grafana recomendados: Istio Mesh Dashboard, Istio Service Dashboard, Istio Workload Dashboard, Istio Control Plane.

Trazas distribuidas — Jaeger

Istio propagates automáticamente los headers de trace context B3 y W3C TraceContext a través del ztunnel y del waypoint proxy. Cada salto de tráfico (gateway → waypoint → pod) genera un span hijo vinculado a la traza raíz.

Jaeger almacena y visualiza las trazas, permitiendo identificar qué microservicio introduce latencia en flujos multi-servicio como reserva → pago → notificación. La integración se configura en el MeshConfig de Istio con extensionProviders apuntando al colector Jaeger.

Kiali — Topología del service mesh

Kiali provee una vista gráfica del service mesh en tiempo real: grafo de dependencias entre microservicios, estado de salud por servicio, tráfico en req/s y tasa de error, y visualización de las AuthorizationPolicies activas.

En modo ambient, Kiali muestra los ztunnels y el waypoint como nodos del grafo, permitiendo ver qué tráfico pasa por políticas L7. Resulta especialmente útil para auditar las reglas DENY del login-handler-sa y verificar que el aislamiento de servicios funcione como se diseñó.

Separación de nodos — Terraform EKS Node Groups

workloads-ng
Node group para los 7 microservicios FastAPI. Instancia t3.medium, min 1 / max 2 nodos. Label role=workloads. Sin taint — todos los pods de negocio pueden desplegarse.

observability-ng

t3.small · 1–2 nodes
Zero-trust con observabilidad completa: la combinación de mTLS automático vía ztunnel, AuthorizationPolicies aplicadas en el waypoint, access logging mesh-wide y métricas/trazas/logs en el stack de observabilidad provee visibilidad y control total del tráfico sin modificar una sola línea de código de los microservicios.
